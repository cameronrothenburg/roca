# Roca Ownership Model: const Borrows, let Moves

## The Problem

Most languages choose between two extremes for memory:

- **Garbage collection** (JS, Go, Java): automatic but unpredictable. GC pauses, non-deterministic cleanup, hidden memory growth.
- **Manual memory** (C, C++): precise but dangerous. Use-after-free, double-free, leaks.

Rust introduced a third path: **ownership checked at compile time**. No GC, no manual free, no leaks. The compiler proves your code is safe before it runs.

But Rust's model has a cost: lifetime annotations, borrow checker fights, `&mut` vs `&` vs owned semantics. This complexity is justified for systems programming, but it's a barrier for AI-generated application code.

Roca takes the core insight from Rust and simplifies it down to two keywords.

## The Model

```
const = immutable, borrowed when passed
let   = mutable, moved when passed
```

That's it. No `&`, no `&mut`, no lifetime annotations, no `Box<T>`, no `Rc<T>`. Two keywords, two behaviors.

### const: Borrow

A `const` binding is immutable. When you pass it to a function, the function borrows it — reads it without taking ownership. The caller keeps the value.

```roca
const name = "hello"
greet(name)           // borrowed — greet reads it
log(name)             // still valid
```

This is safe because `const` values can never be mutated. Multiple functions can read the same data simultaneously without conflicts. No reference counting needed — the creator owns it, everyone else borrows.

### let: Move

A `let` binding is mutable. When you pass it to a function, ownership moves — the caller can no longer use it.

```roca
let items = [1, 2, 3]
process(items)        // moved — process owns it now
log(items)            // COMPILE ERROR: use-after-move
```

This prevents the two most dangerous memory bugs:
- **Use-after-free**: you can't use `items` after it's moved
- **Double-free**: only one owner exists at any time — whoever holds it frees it

### Return Transfers Ownership

Functions can return values to give ownership back:

```roca
let data = [3, 1, 2]
let sorted = sort(data)   // data moves in, sorted result comes back
log(sorted)                // OK — sorted is the new owner
```

The chain of ownership is clear: `data` → `sort()` → `sorted`. At every point, exactly one binding owns the memory.

## Why Not Rust's Full Model?

Rust adds three more concepts on top of ownership:

1. **References** (`&T`, `&mut T`): borrowed pointers with rules about aliasing
2. **Lifetimes** (`'a`): annotations proving how long borrows live
3. **The borrow checker**: static analysis ensuring references are valid

These are powerful but complex. In Roca's world — where code is generated by AI for web services, APIs, and business logic — the full borrow checker is unnecessary overhead.

Roca's simplification works because:

- **`const` is the default.** Most data is immutable. Immutable data can be freely shared without ownership concerns.
- **Functions are mostly pure.** They take inputs, produce outputs. They don't hold references to caller data.
- **Roca has crash blocks.** Error handling is structural, not exceptional. There's no `try/catch` unwinding that could invalidate references mid-scope.

## Compiler Enforcement

The compiler catches ownership violations at compile time:

### use-after-move
```roca
let x = "hello"
consume(x)            // x moved
log(x)                // ERROR: use-after-move
```

### move-in-loop
```roca
let data = "payload"
while condition {
    consume(data)     // ERROR: data moved on iteration 1, gone on iteration 2
}
```

### must-be-const
```roca
let x = 5             // WARNING: x is never mutated — use const
return x
```

### recursive-cycle
```roca
struct Bad {
    self_ref: Bad      // ERROR: infinite size — use Optional<Bad>
}{}
```

## Native Engine: Real Memory Management

The JS target doesn't need memory management — V8's GC handles it. But the native engine (Cranelift JIT/AOT) manages memory for real:

- **Strings** use reference counting with an RC header: `[refcount | size | payload]`
- **Scope exit** releases all local heap variables
- **Return values** skip cleanup — ownership transfers to the caller
- **`let` reassignment** releases the old value before storing the new one
- **Loop bodies** clean up per-iteration locals, including on `break`/`continue`
- **Crash `halt`** releases all locals before propagating the error

Every allocation is tracked by a `MemTracker` with atomic counters. Tests assert exact alloc/free counts — no leaks tolerated.

## The Two Rules

Everything in Roca's memory model follows from two rules:

1. **If you can't change it (`const`), sharing is free.**
2. **If you can change it (`let`), only one owner at a time.**

No GC. No manual free. No lifetime annotations. The compiler does the rest.
